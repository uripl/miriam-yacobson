# ניתוח אחסון - אתר מרים יעקובסון

## איך האחסון עובד היום

האתר משתמש ב-3 שירותים שונים לאחסון:

### 1. GitHub Pages (אירוח האתר)
- האתר הסטטי (HTML/CSS/JS) מאוחסן ב-GitHub ומופץ דרך **GitHub Pages**
- **חינם לחלוטין** לריפוזיטוריים ציבוריים
- מגבלה: 1GB לריפוזיטורי, 100GB bandwidth לחודש
- כולל גם כמה תמונות סטטיות בתיקיית `public/images/`

### 2. Firebase Cloud Storage (אחסון קבצים)
- **כל התמונות והמסמכים שמעלים דרך האתר** נשמרים ב-Firebase Storage
- Bucket: `miriam-yacobson.firebasestorage.app`
- נתיבי אחסון:
  - `gallery/{timestamp}_{filename}` - תמונות הגלריה
  - `documents/{timestamp}_{filename}` - מסמכים (תמונות + PDF)
  - `content/{contentKey}/{filename}` - תמונות תוכן בפרקים

- **מגבלות העלאה בקוד:**
  - תמונות גלריה: עד **10MB** לקובץ
  - מסמכים: עד **25MB** לקובץ
  - תמונות תוכן: עד **10MB** לקובץ

### 3. Firebase Firestore (מסד נתונים)
- שומר **מטא-דאטה בלבד** (כותרות, תיאורים, תאריכים, קישורים לקבצים)
- לא שומר את הקבצים עצמם
- collections: `gallery`, `documents`, `videos`, `content`, `admins`, `members`

### 4. YouTube (סרטונים)
- הסרטונים **לא מאוחסנים ב-Firebase** - רק קישורי YouTube נשמרים ב-Firestore
- **אין עלות אחסון** על סרטונים

## תחזית עלויות

### תרחיש קטן (50 תמונות + 20 מסמכים)
- ממוצע תמונה: ~2MB, ממוצע מסמך: ~5MB
- אחסון: ~200MB (תמונות) + ~100MB (מסמכים) = **~300MB**
- **נכנס בתוכנית החינמית** (5GB)

### תרחיש בינוני (200 תמונות + 100 מסמכים)
- אחסון: ~400MB + ~500MB = **~900MB**
- **עדיין נכנס בתוכנית החינמית**, אבל מתקרבים למגבלה

### תרחיש גדול (500 תמונות + 300 מסמכים)
- אחסון: ~1GB + ~1.5GB = **~2.5GB**
- **עדיין בתוכנית החינמית** (5GB)

### מתי תתחילו לשלם?
- **אחסון**: כשעוברים 5GB (בערך 500+ תמונות באיכות גבוהה + מאות מסמכים)
- **הורדות**: כשיותר מ-1GB הורדות ביום - זה יותר סביר שיקרה! אם יש 100 תמונות בממוצע 2MB, ו-50 מבקרים ביום שטוענים את כל הגלריה = ~10GB/יום

### סיכון עיקרי: **Bandwidth (הורדות)**
**המגבלה הכי קריטית היא 1GB הורדות ביום.** אם אתר עם הרבה תמונות מקבל תנועה, אפילו מתונה, אפשר לחרוג מזה.

לדוגמה:
- 30 תמונות * 2MB ממוצע = 60MB לטעינת גלריה מלאה
- 1GB / 60MB = **~16 מבקרים ביום** לפני שחורגים מהמגבלה

## אלטרנטיבות אחסון

### אופציה 1: Cloudinary (מומלץ לתמונות)
- **תוכנית חינמית**: 25GB אחסון + 25GB bandwidth לחודש
- יתרון גדול: **אופטימיזציית תמונות אוטומטית** (resize, WebP, lazy loading)
- יכול להקטין את גודל התמונות ב-50-80% ללא אובדן איכות נראה
- API פשוט להעלאה ישירה מ-React
- **עלות מעבר לחינמי**: מ-$89/חודש

### אופציה 2: Supabase Storage
- **תוכנית חינמית**: 1GB אחסון, 2GB bandwidth
- שדרוג: $25/חודש ל-100GB אחסון + 250GB bandwidth
- יתרון: ממשק דומה ל-Firebase, מעבר קל

### אופציה 3: AWS S3 + CloudFront
- אחסון: $0.023/GB/חודש (דומה ל-Firebase)
- CloudFront CDN: $0.085/GB bandwidth (זול מ-Firebase)
- יתרון: זול מאוד בסקייל, CDN מהיר
- חיסרון: מורכב יותר להגדרה

### אופציה 4: להישאר עם Firebase + לבצע אופטימיזציות
- **לדחוס תמונות לפני העלאה** (client-side) - להקטין מ-2-5MB ל-200-500KB
- **להשתמש ב-CDN חיצוני** (כמו Cloudflare) לפני Firebase Storage
- **lazy loading** - לטעון תמונות רק כשהן נראות על המסך

## המלצה

**לטווח הקרוב (עד ~200 תמונות ותנועה נמוכה):**
להישאר עם Firebase, אבל **לבצע אופטימיזציית תמונות בצד הלקוח** לפני העלאה (resize + compression). זה הדבר היחיד שיש לו את ההשפעה הכי גדולה על עלויות.

**לטווח הבינוני/ארוך:**
לשקול מעבר ל-**Cloudinary** עבור תמונות ומסמכים. Cloudinary נותן bandwidth הרבה יותר גדול בחינם (25GB/חודש vs 1GB/יום), ויש לו אופטימיזציה אוטומטית שתחסוך 50-80% מגודל הקבצים.

Firestore לא צריך להשתנות - הוא משמש רק למטא-דאטה וצריכת המשאבים שלו זניחה.
